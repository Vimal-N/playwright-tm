1. HTML Fundamentals & Document Structure

Every web page is built on HTML. Understanding document structure helps you navigate the DOM efficiently during test automation.

<!-- Document Type Declaration --> <!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <title>Page Title</title> </head> <body> <!-- All visible content --> </body> </html>
<!DOCTYPE html>
Document Type Declaration

Purpose: Tells browser to use HTML5 standards mode. Missing DOCTYPE causes quirks mode with inconsistent rendering.

<html>
Root Element

Purpose: Root container. The lang attribute specifies document language for screen readers and localization.

<head>
Metadata Container

Purpose: Contains metadata: charset, viewport, title, stylesheets. Not visible but <title> is testable.

<body>
Visible Content Container

Purpose: Contains all visible page content. All UI testing happens here.

Playwright Usage
// Get page title const title = await page.title(); // Check language attribute const lang = await page.locator('html').getAttribute('lang');

2. Text Content & Typography Tags

Heading Tags (h1 - h6)

<h1> through <h6>
Content Hierarchy Headings

Purpose: Define hierarchical structure. h1 = main title (unique per page), h2 = sections, h3-h6 = subsections.

<h1>Main Page Title</h1> <h2>Section</h2> <h3>Subsection</h3>
Playwright Usage
await page.getByRole('heading', { name: 'Dashboard' }).click(); await page.getByRole('heading', { level: 1 }).textContent();

Paragraphs and Inline Text

<p>
Paragraph Block

Purpose: Wraps text as a logical paragraph with margins for visual separation.

<span>
Inline Container

Purpose: Generic inline container for styling or JS hooks. No semantic meaning.

Legacy Issue: Often overused for layout instead of semantic tags.

Text Formatting

Links

<a>
Anchor/Hyperlink
<a href="https://example.com" target="_blank">External</a> <a href="/dashboard">Internal</a> <a href="#section">Anchor</a> <a href="/file.pdf" download>Download</a>
AttributePurposeTesting Note
hrefURL destinationVerify navigation

await page.getByRole('link', { name: 'Dashboard' }).click(); // Handle new tab const [newPage] = await Promise.all([ page.waitForEvent('popup'), page.getByRole('link', { name: 'External' }).click() ]);

Lists

<ul>, <ol>, <li>
List Structures
  • <ul> - Unordered (bullet) list
  • <ol> - Ordered (numbered) list
  • <li> - List item (must be child of ul/ol)
Playwright Usage
await page.getByRole('listitem').filter({ hasText: 'Settings' }).click(); await expect(page.locator('ul li')).toHaveCount(5);

3. Semantic Layout Tags (Modern HTML5)

âœ¨ Why Semantic Tags Matter for Testing

Semantic tags provide meaningful hooks. Instead of div.wrapper > div:nth-child(2), use main article or role-based selectors.

<header>
Page/Section Header â€¢ ARIA role: banner

Purpose: Contains introductory content - navigation, logo, search bar.

Playwright Usage

<nav>
Navigation Section â€¢ ARIA role: navigation

Purpose: Contains navigation links - menu, breadcrumbs, pagination.

<nav aria-label="Main Navigation"> <a href="/">Home</a> <a href="/about">About</a> </nav>
Playwright Usage
await page.getByRole('navigation', { name: 'Main Navigation' }) .getByRole('link', { name: 'Home' }).click();
<main>
Main Content Area â€¢ ARIA role: main

Purpose: Primary unique content. Only one per page.

Playwright Usage
// Scope to main (avoids header/footer) await page.getByRole('main').getByRole('button', { name: 'Submit' }).click();
<section>
Thematic Section

Purpose: Thematic grouping with a heading (Features, Pricing, Contact).

<article>
Self-Contained Content â€¢ ARIA role: article

Purpose: Standalone content (blog post, news, product card).

<aside>
Sidebar Content â€¢ ARIA role: complementary

Purpose: Tangential content - sidebars, pull quotes, ads.

<footer>
Page Footer â€¢ ARIA role: contentinfo

Purpose: Footer info - copyright, contact, sitemap links.

4. Legacy Layout Patterns

âš ï¸ Legacy Pattern Recognition

Common in pre-HTML5 apps. Not "wrong" but offer less semantic meaning for testing.

<div>
Generic Block Container
Legacy Heavy

Problem: "Div soup" - nested divs for all layout instead of semantic tags.

<!-- Legacy "Div Soup" --> <div class="header"><div class="nav">...</div></div> <!-- Modern --> <header><nav>...</nav></header>
Strategy for Div Soup
// Rely on classes await page.locator('div.nav-item').filter({ hasText: 'Home' }).click(); // Or test IDs await page.locator('[data-testid="nav-home"]').click();
<iframe>
Inline Frame
Still Used

Uses: Third-party embeds (YouTube, maps), payment forms (PCI), ads.

Playwright Strategy
const frame = page.frameLocator('iframe[name="payment"]'); await frame.locator('#card-number').fill('4111111111111111');

Legacy vs Modern Quick Reference

LegacyModernPlaywright
div.header<header>getByRole('banner')
div.nav<nav>getByRole('navigation')
div.main<main>getByRole('main')
div.footer<footer>getByRole('contentinfo')
div onclick<button>getByRole('button')

Dropdown Selection

<textarea>

Clickable Button

Form Testing Checklist
  • All inputs have associated labels
  • Required fields marked with 'required'
  • Validation messages display correctly
  • Tab order follows logical flow
  • Submit button has type="submit"
  • File inputs have proper 'accept' attributes

11. Influencing Design & Development Decisions

As a QA engineer, you can advocate for better HTML that improves accessibility AND testability.

Key Discussion Points

1. Request Semantic Elements

Instead of: "This is hard to test"

Say: "If we use <button> instead of <div onclick>, screen readers work AND our tests are more stable."

2. Advocate for Proper Labels

Instead of: "I can't find this input"

Say: "Adding <label for="email"> lets us use getByLabel() AND improves accessibility."

3. Suggest Test IDs When Needed

Instead of: "No stable selector"

Say: "Can we add data-testid here? It won't affect styling but gives us a stable hook."

4. Frame Accessibility as Testing Issues

Instead of: "Playwright can't find this"

Say: "This element has no role or accessible name. That's both an accessibility bug AND makes it untestable."

Pre-Development Checklist

HTML Review Checklist
  • Semantic elements used (header, nav, main, section)?
  • Interactive elements have roles (button, link)?
  • Form inputs properly labeled?
  • Images have meaningful alt text?
  • Headings are hierarchical (h1 â†’ h2 â†’ h3)?
  • Modals use dialog semantics?
  • Custom components have ARIA?
  • data-testid planned for dynamic content?
âœ… Benefits of Good HTML
  • Faster test writing: Semantic selectors are self-documenting
  • Stable tests: Role-based selectors survive CSS changes
  • Accessibility: Meets WCAG compliance
  • Easier debugging: Clear structure = obvious issues
  • Future-proof: Works with new tools and browsers
